# 問題與修復回顧（2026-07-10 ～ 07-13）

Citation skill 重構後連續三天的 CLI 實測，暴露出一批互相牽連的問題。本文整理這段時間遇到的問題、根源與對應改動；細節見 `prob.md`（實測問題清單）、`deep-research-report.md`（問題分群與優先序分析）、`CITATION_UPGRADE_PLAN.md`（批次儲存升級計畫）。

## 一、回合以空回應結束，看起來像「中斷」

**問題**：一個 session 內多次出現工具成功返回後 assistant 毫無輸出、turn 直接結束；使用者誤以為 CLI 卡住，連打的 `y` 還被當成新的對話輪吃掉。

**根源**：兩層疊加。上游 OpenRouter 會間歇性回傳空回應（只有一個結束符、零內容）；而舊的 graph 與 CLI 都接受空字串當最終答案，空回合照樣寫入紀錄、畫面上什麼都沒有。

**改動**：模型 adapter 改為 pinned ChatOpenRouter；每個回應階段記錄可診斷的 telemetry；空回應改為**用同一請求立即重打（最多兩次）**，重試耗盡後回傳確定性的誠實訊息，明說上游回空、本輪未執行任何操作。

## 二、模型瞎掰：捏造儲存成功、腦補內部實作

**問題**：實測中模型在沒有呼叫任何工具的情況下，回報「c4 已成功儲存、BibTeX 已寫入」——實際上檔案根本不存在。也曾腦補 workflow 內部實作（宣稱由模型生成 BibTeX、引用只存在記憶體），或只憑標題就寫出像摘要的論文介紹。

**根源**：舊的空回應救援路徑會禁止工具、再要模型「直接寫出最終回答」——對一個需要工具才能完成的請求，模型只能編。另外 SKILL.md 缺少公開契約（資料血緣、儲存位置、何時必須標示推測），模型便以常見系統模式自行補完。

**改動**：真正空回應不再進入禁工具的 repair（見上一節）；SKILL.md 補上 grounding 規則（metadata-only 必須明示推測）、儲存位置契約，並新增唯讀 explain action 回傳確定性的流程說明。citation gate 持續把關：引用標記必須對應已驗證來源，捏造的 cite 一律封鎖。

## 三、批次儲存做不到、部分失敗被遮蔽

**問題**：一次存三篇論文要被確認三次，最後只存成功一篇，另外兩篇靜默失敗且完全沒被回報。

**根源**：coordinator 是單一 selection 設計（select 會作廢其他候選的 match、confirm 成功就清空整個池子），加上「confirm 必須晚 select 一輪」的硬規則，批次儲存在狀態機層面不可能；finalizer 又只在有成功收據時才渲染，失敗被吞掉。

**改動**：select/confirm 支援批次 identifiers、match 跨候選累積；使用者的請求本身即構成儲存授權（模型判斷語意，同輪可完成 select→confirm；否定/條件/疑問語氣仍禁止）；收據改為批次 artifact，成功與失敗都由 finalizer 確定性渲染，全數失敗也照實列出。

## 四、儲存位置找不到

**問題**：使用者預期 bundle 存在專案根目錄的 `cite/`，實際卻散落在平台 user-data 目錄（`~/.local/share/...`）。

**根源**：早期版本預設寫入平台 user-data 目錄；07-12 下午才改為 workspace 根目錄。之前 session 留下的 bundle 停在舊位置。

**改動**：預設輸出改為 workspace 根目錄 `cite/`（不存在就建立、納入版控），config/env 覆寫優先序保留，非 repo 環境仍有平台目錄作 fail-safe。

## 五、Gate 把成功說成失敗、收據跨輪遺失

**問題**：confirm 實際成功寫入，但同輪回覆因含裸 DOI 被 gate 整段換成封鎖訊息，成功事實消失；下一輪模型只看得到「被封鎖」，於是編出自相矛盾的解釋。

**根源**：gate 與 budget 這類安全機制缺「失敗出口」——封鎖時直接丟棄整份草稿，已提交的副作用沒有留下可信紀錄。

**改動**：confirm 收據走可信 artifact 通道，被封鎖或失敗的回合由 finalizer 確定性渲染收據與 pending 狀態；工具額度耗盡後的協定文字洩漏（把工具呼叫寫成普通文字）也在 finalization 攔截。

## 六、檢索品質與額度不匹配

**問題**：候選池 50 筆、每頁 10 筆，但每輪工具額度只有 4 次，逐頁掃描天然超額；年份範圍被模型改述、venue 品質混排、同作品多版本佔多個候選。

**改動**：新增 refine action 對既有候選做條件精煉；搜尋回應回帶實際套用的年份 filter 且要求照實轉述；建立 venue catalog 做確定性分層標註；版本群組非破壞性呈現。檢索品質仍列為持續觀察項。

## 待觀察

- OpenRouter 回空的上游根因（哪家 provider、什麼條件觸發）尚未定位；目前靠重試緩解，telemetry 已可累積證據。
- 模型是否仍會把批次操作拆成平行工具呼叫。
- Discovery 品質（相關性、重複、分層）屬品質層優化，排在穩定性之後逐步調整。

## 核心教訓

安全機制（gate、budget、救援路徑）都需要明確的失敗出口：封鎖要保留已發生的事實，耗盡要給確定性訊息，上游異常要誠實回報——靜默空白與讓模型自由補完，都會製造比原問題更糟的失效模式。
