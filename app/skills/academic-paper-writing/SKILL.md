# Academic Paper Writing

## When to use

Use this skill when the user is writing or revising scholarly work, including:

- journal articles
- conference papers
- thesis or dissertation chapters
- literature reviews or review articles
- abstracts, titles, keywords, or cover letters
- point-by-point responses to reviewers

Do not treat all disciplines the same. Empirical STEM papers, qualitative studies, legal scholarship, and humanities argument papers follow different conventions. Start from the user's field, venue, and material, then adapt.

## Core rules

1. Read before writing.
   - Read the user's draft, notes, outline, data description, rubric, reviewer comments, and target-journal guidelines with the file-reading tool before proposing major rewrites.
2. Establish the writing target early.
   - Identify the paper type, discipline, target venue, language, word limit, citation style, and current stage.
3. Do not invent scholarly content.
   - Do not fabricate citations, quotations, page numbers, DOIs, data, statistics, ethics approvals, author contributions, reviewer comments, funding statements, or research findings.
4. Prefer argument structure before sentence polishing.
   - If the manuscript is early-stage or structurally weak, fix the claim, section logic, evidence map, and paragraph order before line editing.
5. Keep evidence and inference separate.
   - Mark what is directly supported by the user's supplied sources or draft, and what is a suggested interpretation or rewrite.
6. Respect local conventions.
   - Follow the target journal, department, advisor, or call-for-papers instructions over generic writing advice.
7. Default to concise, explicit prose.
   - Use direct claims, clear paragraph openings, specific verbs, and limited jargon unless the field convention requires dense technical language.
8. Use the relevant reporting or style standard when applicable.
   - If the user names a guideline, or the paper type clearly implies one, follow it.

## Intake checklist

If key constraints are missing, ask only the smallest set of questions needed to avoid bad assumptions. Otherwise proceed and state assumptions explicitly.

Collect or infer:

- research question or thesis
- paper type: empirical article, review article, literature review section, thesis chapter, commentary, response letter
- discipline and subfield
- target venue or rubric
- language
- required citation style
- word limit or page limit
- current draft status: outline, rough draft, revise-and-resubmit, proofreading
- available materials: notes, PDFs, figures, tables, reviewer comments, journal guidelines

## Default process

1. Diagnose the task.
   - Decide whether the user needs planning, drafting, restructuring, editing, submission support, or reviewer-response support.
2. Extract the paper's core message.
   - Write one sentence for the main contribution, claim, or research question before editing the full text.
3. Build a section-level map.
   - Check whether each section has a distinct job and whether claims appear in a logical order.
4. Improve the highest-leverage issue first.
   - Usually this means fixing argument flow, missing context, unsupported claims, or weak synthesis before grammar.
5. Produce the requested output in the format below.
6. Run a final scholarly integrity check.
   - Verify that no citation, claim, or method detail was introduced without support.

## Output formats

### Planning mode

Return:

- a one-sentence thesis or contribution
- a section-by-section outline
- an evidence map showing what each section must prove
- a short list of missing inputs or likely weak spots

### Drafting mode

Return the requested prose directly, but keep it bounded to the evidence the user supplied. If evidence is missing, use bracketed placeholders such as `[insert citation]`, `[report sample size]`, or `[state limitation]` instead of inventing content.

### Revision mode

Return:

- the main problems you found, ordered by importance
- a revised version of the text or section
- a short note on what changed structurally

### Submission support mode

Return the requested artifact directly, such as:

- title options
- abstract
- keywords
- cover letter
- reviewer response table
- submission checklist

## Edge cases

- If the user asks for a full paper without sources, write an outline, template, or placeholder draft rather than fabricating scholarship.
- If the user asks for citations but provides no source set, ask for sources or clearly label any suggested search terms as uncited leads rather than references.
- If the user asks for proofreading only, preserve technical meaning and flag any substantive logic problems separately instead of silently rewriting claims.
- If the user asks to hide AI use, falsify authorship, conceal conflicts, or disguise plagiarism, refuse that part and redirect to an ethical alternative.
